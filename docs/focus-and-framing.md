# Focus & Framing

> The rule for **every** WareFleet build decision, and how to present the project so a robotics professor takes it seriously. Read with [`NORTH-STAR.md`](NORTH-STAR.md).

## Three layers, three roles
| Layer | Role | Investment |
|---|---|---|
| **ROS 2 / Nav2 / Gazebo** (robotics) | **Credibility gate** — must genuinely WORK or the project is dismissed as "backend cosplaying as robotics." Bar = *real*, not novel. My weakest area; **don't try to win here.** | ~35% |
| **Go fleet manager / coordination / telemetry** (systems) | **Differentiator** — where I actually win. Almost no applicant brings distributed-systems + telemetry depth. Go DEEP. | ~45% |
| **Measurement / benchmark / finding** (science) | **Researcher signal** — proves I can do science, not just build. Non-negotiable. | ~20% |

## Failure modes to avoid
- **All robotics, no systems depth** → "another student who did the Nav2 tutorial." Forgettable.
- **All Go, no real robots** → "a backend engineer who doesn't do robotics." Dismissed by robotics profs.
- **✅ Win:** credible robotics substrate + deep systems/coordination + a measured finding.

## The presentation rule
Always frame the software as **robotics-applied**: *"fault-tolerant coordination for autonomous mobile robot fleets"*, never *"a distributed task scheduler."* Lead a demo/email with **robots moving on screen**, then reveal the systems depth underneath.

## Per-professor nuance
- **Défago / systems / multi-agent (Ota):** lean into the software/coordination — it *is* their research.
- **Navigation / SLAM (Miura):** show more visible robotics depth; systems is the bonus.
- **Universally: the robots must run.** No exceptions.

## Bottom line
> **Robotics is the gate you pass; software is the moat you win on; the measurement is proof you're a researcher.**
Build robotics to "credibly works," go deep on Go/coordination + telemetry, always present it as robotics.
